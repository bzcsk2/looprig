# Loop Worker Report 整改计划

## 结论

不要在 `ba41376` detached HEAD 上继续改。这个版本适合当作行为基线和回归参照，但后续修复应先切到当前目标开发分支，再把整改按小步提交实现。

建议顺序：

1. 确认目标分支。
2. 确保工作区干净，避免把 detached HEAD 上的验证改动混进实现提交。
3. 在最新分支上先补回归测试，再改实现。
4. 每个阶段跑对应测试，最后跑全量 `bun test packages/core packages/tui packages/cli` 和 `bun run typecheck`。

## 问题描述

当前需要修复的是 loop 模式中 Supervisor 无法稳定收到 Worker 报告，导致 Supervisor 无法基于真实 Worker 结果做 review / continue / approve / blocked 决策。

目标行为应该是：

1. Supervisor 在 `supervisor_analyse` 只负责产出 plan。
2. Coordinator 把 plan 传给 Worker。
3. Worker 在 `worker_do` 执行任务。
4. Worker 在 `worker_report` 产出报告。
5. Coordinator 把 report 传给 Supervisor。
6. Supervisor 在 `supervisor_check` 必须看到本轮 Worker report，再做判断。

mailbox 版本的代码可以保留，但 loop 主路径先不要依赖 mailbox。主路径应只依赖 coordinator state。

## ba41376 做对了什么

`ba41376` 的关键点是固定编排简单：

1. `runSupervisorAnalyse()` 拿到 Supervisor plan 后调用 `setSupervisorPlan(plan)`。
2. 随后直接 `transition("worker_do")`。
3. `runWorkerDo()` 使用 `this.state!.supervisorPlan` 构造 Worker 输入。
4. `runWorkerReport()` 从 Worker engine state 取最后一条 assistant message，调用 `setWorkerReport(report)`。
5. `runSupervisorCheck()` 使用 `this.state!.workerReport` 构造 Supervisor review 输入。

虽然 `ba41376` 里已经有 mailbox 读写，但 TUI 主路径没有注入 `agentComm`，所以实际运行仍然依靠 coordinator state fallback。这是它能正常调动 Worker 的主要原因。

## 后续版本容易坏的点

需要重点排查这些变更是否破坏 report 传递：

1. mailbox 变成默认主路径后，Worker report 可能写进 mailbox，但 Supervisor 读不到同一个 thread / goal / workflow 下的消息。
2. `workflowId` 和 sessionId 不一致，会导致 goal、mailbox、coordinator state 分裂。
3. bridge 外层自动重跑 `runWorkflow()`，可能在 coordinator 已经进入 `blocked` 或非启动 phase 时再次调用，打断正常阶段流。
4. Supervisor 工具错误被当作 analysis fatal error，导致 plan 已产生但 coordinator 不进入 `worker_do`。
5. Worker report 的提取依赖 `getWorker().getState().messages.findLast(...)`，如果 Worker engine 的上下文被清理、切 session 或 report submit 没落盘，Supervisor 会拿到空 report。

## 整改方案

### Phase 1: 建立回归测试

先补测试，不改实现。

1. `WorkflowCoordinator` 应完整推进：
   `supervisor_analyse -> worker_do -> worker_report -> supervisor_check`。
2. Worker `worker_do` 输入必须包含 Supervisor plan。
3. Worker `worker_report` 的 assistant 内容必须被写入 `state.workerReport`。
4. Supervisor `supervisor_check` 输入必须包含同一轮 `state.workerReport`。
5. 即使 mailbox 中存在旧消息，默认 loop 主路径也不能读取旧 mailbox 内容覆盖 state。
6. `workflowId === sessionId` 时，goalStore / coordinator / bridge 看到的是同一个目标。

### Phase 2: 明确主路径

把 loop 主路径固定为 coordinator state：

1. `runSupervisorAnalyse()`：只写 `state.supervisorPlan`。
2. `runWorkerDo()`：只读 `state.supervisorPlan`。
3. `runWorkerReport()`：只写 `state.workerReport`。
4. `runSupervisorCheck()`：只读 `state.workerReport`。
5. mailbox 代码保留在显式开关中，例如 `useMailboxWorkflow === true`，默认关闭。

### Phase 3: 修正错误处理

Supervisor plan 阶段不应因为非主路径工具错误直接阻断 Worker：

1. 如果 Supervisor 最终产出了非空 plan，`tool_not_allowed` 这类错误应降级为 warning。
2. 只有模型请求失败、没有最终 plan、或真实 fatal error 才能阻止 `transition("worker_do")`。
3. 日志中保留 rejected tool 调用，方便诊断模型越权行为。

### Phase 4: 收紧工具和提示

loop Supervisor 默认只暴露：

1. `get_goal`
2. `update_goal`

不暴露：

1. `read_mailbox`
2. `send_message`
3. `followup_task`
4. `AgentTool`
5. 工程工具如 `bash/edit/write_file`

Worker loop 不应获得 goal/mailbox 工具，只保留配置中的工程工具。

### Phase 5: bridge 恢复单一驱动

bridge 不应该自行改变 coordinator 的阶段语义：

1. `runWorkflow()` 只消费 coordinator event stream。
2. 不在 bridge 里对 active goal 盲目再次调用 `runWorkflow()`。
3. 如果确实需要自动下一轮，只允许从 `idle` 或 `supervisor_analyse` 这类合法启动 phase 重入。
4. `blocked/completed/failed/waiting_user` 都不能自动重入。

### Phase 6: 验证日志

修复后手动跑 loop，日志中必须能看到：

1. Supervisor submit start/done。
2. phase_change 到 `worker_do`。
3. Worker submit start/done。
4. phase_change 到 `worker_report`。
5. Worker report submit start/done。
6. `state.workerReport` 非空。
7. phase_change 到 `supervisor_check`。
8. Supervisor check 输入包含 Worker report。

## 验收标准

1. loop 模式下 Supervisor 不直接执行工程任务。
2. Supervisor plan 后 Worker 必须启动。
3. Worker report 后 Supervisor 必须能看到 report。
4. mailbox 关闭时，旧 mailbox 消息不会影响 loop 主路径。
5. mailbox 代码仍保留，未来可通过显式开关重新启用。
6. `bun run typecheck` 通过。
7. `bun test packages/core packages/tui packages/cli` 通过。
